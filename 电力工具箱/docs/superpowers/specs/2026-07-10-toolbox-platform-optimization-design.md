# 电力工作工具箱平台优化设计

## 1. 背景与目标

当前项目是一个面向 macOS 的多工具工作区，统一 Tkinter 工具箱已接入以下八项功能：

1. 导出上网电量
2. 电力交易分析
3. 电量汇总
4. 生成报告
5. 私有数据上传
6. 上传集团每日数据
7. 市场表更新
8. WPS 写入工具

本轮优化同时覆盖性能、交互界面、可扩展性、可维护性和稳定性。采用渐进式平台重构，不一次性重写稳定业务脚本。

## 2. 硬性兼容约束

- 保留八个工具现有的业务规则和计算口径。
- 保留配置文件、登录状态、浏览器 profile 和锁文件的现有位置与格式。
- 保留 Excel、Word、HTML 等输出内容、格式和默认位置。
- 保留现有独立启动入口以及根目录统一启动入口。
- 保留每个浏览器工具独立的 `auth_state.json`、`.browser-profile` 和锁文件，继续支持并行运行。
- 上传、覆盖和外部写入仍需用户确认，不得在测试中自动执行生产写入。
- 实施时保护工作区已有的未提交改动，只修改计划明确列出的文件。

## 3. 当前问题

### 3.1 交互

- 主窗口按八个平级导航项组织，缺少工作概览、业务分组和常用操作入口。
- 任务状态只存在于各自页面，切换页面后难以同时观察多个运行任务。
- 失败提示以异常文本为主，缺少可操作的恢复建议、结果定位和安全重试入口。
- 每个页面都固定展示日志区域，信息层级和可用空间不够灵活。

### 3.2 性能

- 启动时一次性创建八个页面，并动态加载部分大型业务模块。
- 每行日志都通过一次 Tk `after()` 调度写入，高输出任务可能造成事件队列压力。
- 文件扫描、模块导入和部分输入预检仍可能在主线程执行。
- 动态模块在不同页面操作中可能被重复加载。

### 3.3 架构与维护

- `toolbox/pages.py` 已超过一千行，页面布局、参数校验、业务适配和任务构造集中在同一模块。
- 页面清单、导航和页面工厂之间依赖固定名称，新增工具需要同时修改多个位置。
- 线程、子进程、日志、错误和忙碌状态由页面基类直接协调，扩展能力有限。
- 若干业务脚本超过一千行，但不适合在本轮一次性重写。

### 3.4 稳定性

- 现有任务注册表能够终止子进程，但缺少统一任务状态、取消中状态、超时和结果快照。
- Python 线程没有通用的协作取消协议。
- 后台任务执行、取消、关闭程序和错误恢复的自动化测试覆盖不足。
- 有副作用操作和纯读取任务尚未在平台契约中明确区分。

## 4. 总体方案

采用“工作台 UI → 平台服务 → 工具适配器 → 现有业务脚本”的四层结构。

### 4.1 工作台 UI

工作台负责导航、任务概览、快捷操作、最近结果和环境诊断，不直接了解业务脚本的调用细节。

界面采用三栏结构：

- 左侧是按“数据采集”“分析与报告”“上传与写入”分组的导航。
- 中间是工作台首页或当前工具页面。
- 右侧是跨页面可见的任务中心；窗口空间不足时可折叠。

### 4.2 平台服务

平台服务包括页面注册表、任务引擎、事件与日志队列、子进程运行器和诊断服务。它们提供稳定接口，不依赖具体工具。

### 4.3 工具适配器

每个工具适配器负责：

- 校验页面输入并生成不可变的参数快照。
- 创建统一 `TaskSpec`。
- 构造函数调用或非 shell 子进程命令。
- 定位输出文件或目录。
- 把底层异常翻译为用户可理解的原因和处理建议。
- 声明任务是否可取消、是否具有外部副作用、是否允许重试。

### 4.4 现有业务脚本

业务脚本继续作为可信业务核心。第一阶段只通过适配器调用，不改变业务规则。后续仅在实际维护频繁或测试隔离困难时，将单个脚本内部拆成更小模块。

## 5. 核心数据契约

### 5.1 `ToolDescriptor`

工具描述符至少包含：

- 稳定的工具 ID
- 显示名称
- 业务分组
- 页面工厂
- 工具适配器工厂
- 可选的快捷操作和依赖声明

主窗口只消费工具目录。新增第九个工具时不修改导航循环或任务引擎，只注册一个描述符并提供页面与适配器。

### 5.2 `TaskSpec`

任务规范至少包含：

- 任务 ID、工具 ID 和显示名称
- 参数快照
- 线程函数或子进程命令
- 工作目录和环境变量
- 是否可取消
- 是否有外部副作用
- 超时策略
- 成功结果定位器

### 5.3 `TaskSnapshot` 与事件

任务状态限定为：

- `created`
- `running`
- `cancelling`
- `succeeded`
- `failed`
- `cancelled`

任务通过结构化事件报告日志、进度、阶段、结果和错误。UI 只读取任务快照和事件，不直接读取线程或进程对象。

## 6. 工作台与交互设计

### 6.1 首页

首页展示：

- 等待、运行中、今日完成和失败任务数量。
- 常用工具快捷入口。
- 最近完成结果及“打开文件/目录”。
- 最近失败任务及处理建议。
- 环境诊断入口。

快捷项默认包含生成报告、市场表更新、上传集团数据和 WPS 写入，后续允许用户调整但不在第一阶段实现持久化自定义。

### 6.2 导航

保留八个现有工具名称，按业务分组展示。页面第一次进入时创建，之后缓存页面实例，因此切换页面不丢失输入、日志或页面状态。

### 6.3 任务中心

任务中心跨页面显示所有活动任务和最近结束任务。运行中的任务显示当前阶段、进度、查看日志和取消按钮；失败任务显示简明原因、处理建议和允许时的重试入口。

### 6.4 确认和重试

- 上传、覆盖和外部写入在执行前显示日期、来源文件、目标和覆盖行为摘要。
- 只读任务和纯本地生成任务可根据原参数快照重试。
- 上传、覆盖和写入绝不自动重试；手动重试前必须再次确认。
- 自动重试只用于明确的瞬时读取失败，并设置有限次数和退避。

## 7. 性能设计

### 7.1 按需加载

启动时只创建窗口壳、工作台和任务中心。工具页面在首次访问时创建并缓存。大型动态业务模块在任务开始或首次需要其能力时加载。

### 7.2 日志批处理

后台线程和子进程只向线程安全队列写入事件。Tk 主线程每 50 至 100 毫秒批量取出并更新日志控件。单页保留最近的可见日志行，完整诊断信息仍可保存到本地日志文件。

### 7.3 非阻塞操作

以下操作不得在 Tk 主线程执行：

- 大目录文件扫描
- 大型动态模块导入
- Excel/Word 内容预检
- 网络、Playwright 和 Node 调用
- 大量输出文件存在性检查

### 7.4 模块缓存

动态模块以规范化路径和文件修改时间作为缓存依据。生产运行复用未变化的模块；文件发生变化时允许缓存失效，避免开发和测试读取过期代码。

### 7.5 性能验收

- 先记录当前机器上冷启动到工作台可交互的真实基线，优化后至少缩短 40%。
- 连续写入 10,000 行日志期间仍能切换页面并触发取消。
- 页面首次加载耗时可记录到诊断日志，后续加载使用缓存。
- 若某指标受 Tk 或 macOS 环境限制，最低验收线是相对基线不退化，并保存可解释的测量结果。

## 8. 稳定性与错误处理

### 8.1 取消

- 线程任务使用取消令牌，在文件循环和阶段边界检查并协作退出。
- 子进程任务先发送温和终止；超时后强制结束，并回收 stdout、文件句柄和进程对象。
- 取消子进程后应在 5 秒内完成回收，自动化测试确认没有遗留进程。

无法安全中断的原子写入阶段应临时禁用取消，并在界面明确显示“正在完成写入”。

### 8.2 退出程序

关闭窗口时若有活动任务，显示任务清单并请求确认。确认后统一取消并等待回收；不得扫描或结束工具箱未创建的系统进程。

### 8.3 错误模型

错误分为输入错误、依赖缺失、登录失效、瞬时网络错误、权限/文件占用、业务数据错误、外部命令失败和用户取消。界面展示简明说明与建议，日志保留原始 traceback、命令退出码和关键上下文。

任何任务结束路径都必须恢复页面和按钮状态。单个任务失败不得破坏其他页面或任务。

## 9. 目标模块结构

```text
toolbox/
├── app.py
├── catalog.py
├── diagnostics.py
├── tasks/
│   ├── models.py
│   ├── engine.py
│   ├── process.py
│   └── events.py
├── ui/
│   ├── dashboard.py
│   ├── navigation.py
│   ├── task_center.py
│   └── theme.py
├── pages/
│   ├── base.py
│   └── eight focused page modules
└── adapters/
    └── eight focused adapter modules
```

`pages.py` 迁移期间保留兼容导出，避免现有导入者立即失效。完成迁移并确认没有消费者后，再删除兼容层。

## 10. 测试策略

### 10.1 回归基线

当前工具箱测试基线为 84 项通过、2 项因环境跳过。实施前后均从 `电力工具箱` 目录运行完整测试。

### 10.2 单元测试

覆盖：

- 任务状态转换
- 取消和超时规则
- 参数快照
- 错误分类和建议
- 工具注册与分组
- 输入验证和输出定位

### 10.3 Tk 组件测试

覆盖：

- 工作台和导航创建
- 页面按需创建且只创建一次
- 页面状态在切换后保留
- 任务中心状态更新
- 日志批量刷新和行数限制
- 窗口关闭确认

### 10.4 集成测试

使用可控的短期子进程测试实时输出、成功、非零退出、取消、超时和回收。不得使用真实生产上传验证任务引擎。

### 10.5 安全冒烟

每个阶段执行：

- 完整工具箱单元测试
- 变更 Python 文件的语法编译
- `TOOLBOX_SMOKE=1 ./电力工具箱.command`
- 对应的启动或性能基准
- 不产生生产副作用的真实文件生成或只读检查

## 11. 分阶段实施

### 阶段一：稳定性底座

实现任务模型、事件队列、日志批处理、线程取消、子进程回收、错误分类和相关测试。现有页面通过兼容接口接入，界面外观暂不大改。

### 阶段二：工作台界面

实现主题、业务分组导航、工作台首页、任务中心、结果卡片、确认摘要和诊断入口。

### 阶段三：性能

实现页面懒加载、动态模块缓存、非阻塞预检和性能基准。记录优化前后数据，不以主观感受替代测量。

### 阶段四：模块拆分

将 `pages.py` 拆成页面模块与适配器，建立工具目录注册机制。之后根据维护频率和风险逐个拆分大型业务脚本，不在本阶段一次性重写全部业务模块。

每个阶段都必须产生可运行、可测试、可回退的结果，并在完整验证通过后再进入下一阶段。

## 12. 非目标

- 不更换 Tkinter 或引入新的大型 GUI 框架。
- 不改变现有业务计算口径、上传目标或报表格式。
- 不引入远程服务、账号系统、云端同步或自动更新。
- 不在第一阶段实现用户自定义仪表盘布局或持久化快捷项。
- 不为了降低文件行数而一次性重写所有大型业务脚本。
